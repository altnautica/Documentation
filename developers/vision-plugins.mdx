title: "Vision plugins"
description: "Use the agent's vision model registry from a plugin: list, download, and load inference models."
---

The agent ships a vision **model registry**: a service that fetches a model
catalog, downloads the right variant for the board, and caches the weights on
disk. A plugin uses the registry to get a model onto the device and then runs
its own inference against the frames it captures.

There is no host-owned frame bus or shared inference loop. Each plugin that
needs computer vision opens its own camera or sensor source, runs its own
model, and acts on the result. The registry exists so plugins do not each
ship and manage their own large weight files.

## What the registry provides

| Surface | What it does |
|---------|--------------|
| `GET /api/vision/models` | Lists the registry catalog, the installed models, and cache usage. |
| `POST /api/vision/models/{model_id}/download` | Downloads a model, picking the best variant for this board. |
| `GET /api/vision/models/{model_id}/status` | Reports download progress and installed state. |

The registry is decoupled from any specific autonomy feature. Any plugin that
needs an inference model loads it through the same registry, so weights are
fetched once and shared across the cache rather than bundled per plugin.

## A worked extension: vision-nav

The first-party `vision-nav` extension is the reference for vision work on the
agent. It provides GPS-denied navigation: optical flow plus monocular visual
inertial odometry, feeding position estimates back to the flight controller. It
runs against ArduPilot 4.5+, PX4 1.14+, and iNav 7.0+, and selects its model
variant through the registry described above.

Look at that extension in `ADOSExtensions/extensions/vision-nav/` for a
real `manifest.yaml`, config schema, and agent/GCS split before you build your
own vision plugin.

## Requesting a model

Declare the model you need in the plugin manifest. On start, ask the registry
to download it (no-op if already cached), then load the weights from the cache
path:

```python
class MyPlugin(Plugin):
    async def on_start(self, ctx: Context) -> None:
        # ensure the model is present (downloads the board-appropriate variant)
        model_path = await ctx.vision.ensure_model("com.example.weed-detector")
        # load it with your own runtime (ONNX Runtime, TFLite, RKNN, ...)
        self.session = load_onnx(model_path)
```

The download picks the variant that fits the board: a quantized build for a
low-power SoC, a full-precision build where there is an NPU or GPU to spare.
You do not choose the variant; the registry does, based on the hardware
fingerprint.

## Running inference

A vision plugin owns its capture and its inference. Open the camera through the
camera driver layer, pull frames, and run them through your loaded model:

```python
async for frame in ctx.camera.frames():
    # frame.width, frame.height
    # frame.format ("rgb24" | "nv12" | "yuv420p")
    # frame.data (bytes)
    detections = self.run(frame)
    for d in detections:
        # act on the detection: steer a gimbal, log it, overlay it
        ...
```

Camera hardware is handled by the
[camera driver layer](/developers/camera-driver). Your plugin reads frames from
the camera it is configured to use; it does not receive frames from a shared
host bus.

## Acting on detections

A detection is only useful if the plugin does something with it. Common actions:

- Drive a gimbal to track a target through the gimbal capability.
- Log detections to the plugin's own store for later review.
- Render an overlay on the recording through the recording overlay surface.

There is no automatic detection bus that fans your results out to other
consumers. The plugin that produces a detection is the plugin that acts on it,
or it publishes through an explicit capability it has been granted.

## Resource discipline

Inference is expensive. A plugin running a 30 MB ONNX model at 30 FPS on a
Rock 5C Lite is fine; running four of them is not. Declare honest resource caps:

```yaml
agent:
  resources:
    max_ram_mb: 512
    max_cpu_percent: 50
```

The supervisor enforces the caps. If your plugin breaches them, the supervisor
throttles or restarts the process. Keep your inference cadence below your CPU
budget and drop frames under load rather than queueing them.

For NPU-based inference on supported boards, load the board's accelerator
runtime directly (TFLite delegates, RKNN, etc.). Using the NPU is the
difference between 50% CPU and 5% CPU on most vision plugins.

## Bundling weights

You can ship small weights as assets in the archive instead of pulling them
from the registry:

```yaml
assets:
  - path: "agent/models/weeds.onnx"
    role: "agent-source"
    sha256: "..."
```

The plugin loads them from the data dir at start. Do not fetch weights from a
remote URL at runtime; the plugin's `network.outbound` policy blocks runtime
fetches by default. Use the registry or a bundled asset.

## Vendor binary models

Closed-source vendor binaries (e.g. proprietary thermal AGC processing as a
`.so`) ship under `vendor/` per the
[vendor binaries](/developers/vendor-binaries) doc. The plugin loads them with
`ctypes` or `cffi` and calls them from its own inference path.
